# “肌肌们”产品与实施规划

## 1. 定位与边界

“肌肌们”是训练完成后的轻社交与收集页，不改变“点身体 → 点表情 → 自动完成”的核心流程。首版包含两个入口：

- **肌友榜**：比较一段时间内的锻炼天数，以及上半身、下半身、全身的肌肉肿大值。
- **暴击图鉴**：从已有训练记录确定性地还原已遇到的暴击倍率和稀有文案。

不增加运动类型、时长、重量、卡路里、目标、积分、提醒或独立登录。`WorkoutRecord` 继续只保存 `bodyArea`、`intensity`、`date`、`createdAt`、`updatedAt`。

## 2. 页面结构

第三个 Tab 命名为“肌肌们”，页面只在用户主动进入时加载社交数据。

### 2.1 顶部概览

- 本周期锻炼天数和自己的排行位置。
- “我的图鉴”收集进度；进度是展示信息，不兑换、不形成积分。
- 未加入肌友时显示轻量空状态：“邀请一个肌友来比比”。

### 2.2 肌友榜

提供周期切换“本周 / 本月”，默认本周；提供四个互斥榜单：

- 锻炼天数：周期内有有效记录的去重日期数。
- 上半身肿大：周期内所有 `upper` 和 `full` 记录对上半身产生的增长值之和。
- 下半身肿大：周期内所有 `lower` 和 `full` 记录对下半身产生的增长值之和。
- 全身肿大：周期内 `full` 记录产生的增长值之和；不把上下半身相加，避免同一条记录被重复包装成另一种指标。

肿大值复用现有 `calculateWorkoutGrowth` 的确定性暴击倍率。榜单展示昵称、头像、数值和并列名次，不展示具体训练日期、难度或单次记录。数值相同时并列，自己的行固定可见。

### 2.3 暴击图鉴

- 按倍率分组：×1.5、×2、×5；普通 ×1 不进入图鉴。
- 每条文案的解锁条件是：历史记录经 `getCriticalMultiplier` 和 `getCriticalQuip` 确定性计算后命中该文案。
- 已解锁项展示文案、倍率和首次命中日期；未解锁项只显示剪影与稀有度，不透露完整文案。
- 图鉴可完全从已有记录重建，不在 `WorkoutRecord` 上保存 `achievementId`、倍率或文案。

## 3. 纯函数与数据模型

训练事实与社交投影必须分离：

```ts
interface LeaderboardSnapshot {
  ownerId: string
  period: 'week' | 'month'
  periodKey: string
  activeDays: number
  upperGrowth: number
  lowerGrowth: number
  fullGrowth: number
  calculatedAt: string
  schemaVersion: 1
}

interface MuscleFriendship {
  memberIds: [string, string]
  status: 'pending' | 'accepted' | 'blocked'
  createdAt: string
  updatedAt: string
}
```

`ownerId` 是服务端持有的内部身份标识，不暴露给其他客户端。用户昵称和头像仍属于用户资料，不复制到训练记录。

建议增加以下纯函数，并以单元测试固定口径：

- `periodRange(asOf, period)`：按北京时间产生自然周（周一开始）或自然月范围。
- `summarizeLeaderboard(records, range)`：从记录得到四项榜单汇总。
- `rankEntries(entries, metric)`：稳定排序、并列名次和本人定位。
- `deriveCriticalCollection(records)`：确定性生成图鉴解锁状态与首次命中日。

页面只通过独立的 `SocialRepository` / `AchievementRepository` 接口读取数据，不直接查询集合；图鉴仓库首版可由 `WorkoutRepository.list` 加纯函数组成。

## 4. 数据与隐私边界

### 4.1 默认私有

- 原始 `workouts` 保持仅本人可读写，不能为排行榜改成所有用户可读。
- 用户没有建立肌友关系时，不发布、也不查询任何排行汇总。
- 建立关系前明确说明会共享的只有：头像、昵称、周期锻炼天数和三个肿大汇总。
- 不共享单次难度、具体锻炼日、`createdAt`、`updatedAt`、连续天数或历史明细。

### 4.2 可撤回与最小保留

- 用户可解除或屏蔽肌友；操作后双方立即不能再读对方汇总。
- 排行快照只保留当前周、上周、当前月、上月，过期自动删除。
- 退出排行榜时删除本人的公开快照，但不删除私有训练记录。
- 排行读取必须在可信服务端校验双方为 `accepted`，客户端不能仅凭对方 ID 查询。

### 4.3 可信度

客户端本地汇总易被篡改，因此正式好友排行应由可信服务端根据私有记录生成或验证快照。若当前约束不允许新增服务端能力，则首版只能做“我的图鉴 / 我的本地预览”，不能把客户端自报数值宣传为可信好友排行。

## 5. 微信好友能力的可行路径

### 5.1 不可直接采用：微信好友关系链

FitPocket 是原生微信小程序。微信的 `getFriendCloudStorage`、`setUserCloudStorage` 和开放数据域好友榜属于**微信小游戏**关系链能力，普通小程序不能借此读取微信好友列表。因此产品文案不应承诺“自动发现微信好友”，也不应索取通讯录。

### 5.2 推荐路径：分享卡片建立双向肌友

1. 用户 A 在“肌肌们”点击邀请，通过小程序分享卡片发送一次性邀请 token。
2. 用户 B 打开卡片，看到共享字段说明并主动接受。
3. 服务端绑定 A、B 的内部身份，token 单次使用且短期过期。
4. 双方成为 `accepted` 后，服务端才允许互读最小化排行快照。

这利用微信的分享入口触达真实好友，但关系本身由双方明确同意建立。可补充“面对面邀请码 / 小程序码”，使用相同 token 机制，不新增另一套关系模型。

### 5.3 约束冲突处理

仓库规则禁止增加云服务，而当前工程已经使用 CloudBase 做私有同步。肌友关系与跨用户排行必然需要受信任的跨用户授权层，不能用“所有用户可读”的数据库规则替代。进入在线排行榜阶段前，需由产品负责人明确批准：是在现有 CloudBase 范围内增加云函数和社交集合，还是暂缓在线社交。未经批准不实施跨用户数据访问。

## 6. 分阶段实施

### Phase 0：口径与安全设计（本阶段）

- 固定页面结构、四项指标、并列规则和北京时间周期。
- 固定分享字段、撤回规则、保留周期和邀请状态机。
- 验证普通小程序无法直接读取微信好友关系链。
- 不修改业务代码和 `WorkoutRecord`。

验收：产品、隐私和技术三方认可口径；明确是否批准现有 CloudBase 内的社交服务端扩展。

### Phase 1：本地图鉴与榜单计算内核

- 实现并测试四个纯函数。
- 增加“肌肌们”Tab 骨架、我的周期概览和暴击图鉴。
- 肌友榜先显示空状态与邀请说明，不制造假好友数据。
- 从 `WorkoutRepository` 读取记录并派生展示，不增加持久化字段。

验收：相同记录在任意设备产生完全相同的倍率、文案、图鉴和榜单汇总；核心训练流程点击数不变。

### Phase 2：邀请与双向关系（需要服务端扩展批准）

- 增加一次性邀请 token、接受、解除和屏蔽流程。
- 增加 `SocialRepository`，页面不直接访问云集合。
- 服务端校验调用身份、关系状态、token 有效期和幂等性。
- 补齐隐私声明和真机双账号权限测试。

验收：未接受、已解除或已屏蔽关系均无法读取对方资料和快照；邀请不能被重放。

### Phase 3：可信好友排行（需要服务端扩展批准）

- 由服务端从私有训练事实生成周期快照，并只向已接受肌友返回。
- 支持本周、本月四榜切换、并列名次、本人固定行和空状态。
- 加入过期快照清理和退出排行数据删除。

验收：客户端伪造数值无效；两个测试账号只能看见互为肌友的汇总；不能由排行反推出具体训练日或难度。

### Phase 4：小范围验证

- 只观察 Tab 到达、图鉴查看、邀请发起/接受、7 日回访等聚合指标。
- 不采集训练明细作为分析事件，不做提醒或目标系统。
- 根据使用情况调整文案与布局，不增加新训练字段。

## 7. 实现顺序建议

下一次代码迭代先做 Phase 1：纯函数、测试和本地 UI。这部分不依赖新的跨用户权限，风险最低，也能独立交付图鉴价值。在线好友排行必须等服务端授权边界被明确批准后再开始。
